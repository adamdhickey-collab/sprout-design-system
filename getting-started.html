Bring Sprout into your project.

This page is for the second half of using a design system — not what things look like, but how to actually build with them. Three parts: how the tokens are structured, how to pull them into your codebase, and what to check before you ship a new component.

How the tokens are structured

Every color, space, radius, and shadow in Sprout is a CSS custom property, layered in two tiers. Primitives are the raw values (--green-600, --space-5) — see the Color and Layout & shape foundations for the full scale. Semantic tokens point at a primitive for a specific purpose (--brand, --surface, --text-muted) and are what components actually use, so a single re-point in [data-theme="dark"] updates every component at once. The full mapping is on the Themes page.

/* Reference a token the same way any component does */
.my-card {
  background: var(--surface);
  border: 1px solid var(--border);
  color: var(--text);
  border-radius: var(--radius-md);
  padding: var(--padding-xl);
}

Adding Sprout to a codebase

looks_oneCopy the source. styles.css, the Cargill Fonts/ folder, and assets/ are the entire dependency — no build step, no package manager. Copy all three into your project.
looks_twoLink the stylesheet and keep the font <link>/<preload> tags from any page's <head> here as a reference — Big Caslon and Helvetica Now won't load without them.
looks_3Copy markup from the source, not the screenshot. Every example throughout this site has a Code toggle in its top-right corner that reveals the exact HTML behind it — copy that instead of rebuilding it from what you see.
looks_4Toggle dark mode before you trust anything. A component that looks right in Light hasn't been verified until you've flipped the switch in the sidebar and looked again.

Before you ship a new component

This checklist exists because of a real bug class found in this system: components that looked correct in Light mode broke under Dark — a wordmark that went invisible, destructive-red text that dropped to 2.9:1 contrast, white text baked onto a fill that turns lime in Dark mode. All from the same root cause below.

toggle_onKnow which tokens re-point in Dark mode. --brand, --brand-deep, --surface, and the --text-* family intentionally change value under [data-theme="dark"]. Most primitives (--green-600, --red-400, --neutral-1000) do not.
lockIf a surface must never change with the visitor's theme — a badge fill, a footer band that's always dark, a floating control that's always a light pill — reach for a fixed primitive, not a semantic token. Using --surface or --brand out of habit is exactly how this bug class happens.
contrastNever hardcode a foreground color against a token background. White text baked onto an element filled with --brand reads fine in Light (a dark green fill) and disappears in Dark (a lime fill). Use --on-accent instead, which flips with the fill.
visibilityCheck every state, in both themes. Default, focused, error, disabled, and skeleton — each one, with the sidebar toggle flipped to Dark. This is the single check that would have caught every bug in this list before it shipped.
